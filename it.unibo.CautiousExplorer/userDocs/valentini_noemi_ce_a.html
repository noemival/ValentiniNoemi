Introduction

This case-study starts to deal with the design and development of proactive/reactive software systems that use aynchronous exchange of information.

Requirements

Design and build a software system that allow the robot described in VirtualRobot2021.html to exibit the following behaviour: the robot lives in a closed environment, delimited by walls that includes one or more devices (e.g. sonar) able to detect its presence; the robot has a den for refuge, located near a wall; the robot works as an explorer of the environment. Starting from its den, the robot moves (either randomly or - preferably - in a more organized way) with the aim to find the fixed obstacles around the den. The presence of mobile obstacles is (at the moment) excluded; since the robot is 'cautious', it returns immediately to the den as soon as it finds an obstacle. Optionally, it should also return to the den when a sonar detects its presence; the robot should remember the position of the obstacles found, by creating a sort of 'mental map' of the environment.

  • closed envirorment: a room delimited by walls;
  • den: start position and refuge near a wall;
  • robot: a device able to execute move commands sent over the network, as described in the document VirtualRobot2021.html provided by the customer;
  • obstacles:are fixed which could be for example a box or walls too, they could be more than one;
  • Optionally: could meaning that the action about the sentence "the robot should also return to the den when a sonar detects its presence" has second priority about the project;

  • Mental Map:it should be a way to represent the moves of the robot;

    Problem analysis

    Port Choose

    Because the robot is the same of boundaryWalk, we highlight some similiar things:
    1. In the VirtualRobot2021.html: commands the customer states that the robot can receive move commands in two different ways:
      • by sending messages to the port 8090 using HTTP POST
      • by sending messages to the port 8091 using a websocket
    2. With respect to the technological level, there are many libraries in many programming languages that support the required protcols. (mental map tools)
      If we ignore the optionally requirement the project could make in a request-response way, with HTTP POST, else it is important to specify that the application to comunicate with the sonar must to use websoket protocol.

      3. Cril vs Aril

      Using aril in an other layer makes the robot a 'logical entity' able to 'talk' with clients in a custom high-level language, it indeed means abstract "robot interaction language" .

    Logical architecture

    We must design and build a distributed system with two software macro-components:
  • the VirtualRobot, given by the customer
  • our cautionExplorer application that interacts with the robot with a request-response pattern
    • It is important to define what the robot has to do the meet the requirements: 
    let us define emum direction {UP,DOWN,LEFT,RIGHT} 
the robot start in the den position, and it need to move random in the closed envirorment: 
    1) send to the robot the request to execute a random (or not, we will see later) command move* and memorized moves,
    until the robot response about "collision"  becomes 'true';
    2) send to the robot the request to execute moves to  back to the den.
    Some considerations about requirements:
    1) "the robot moves with the aim to find the fixed obstacles": for first moves the robot hasn't a "mental map" where it can take informations about where it can found the fixed obstacles.
    2)"it returns immediately to the den as soon as it finds an obstacle": if it doesn't take the same way it could happen that the robot done another collision; so we could problaly define reverse directons.
    3) random or preferibily organized way: if we should to memorized the robot's moves we are talking always about organized way.
