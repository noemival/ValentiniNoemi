Introduction

Requirements

Design and build a ButtonLed software system (bls) that makes human users able to control a led according to seguent control policies each time a button is pressed. Turn on/off a led using a button (ON/OFF). The system should be a distributed system, in which the Button and the Led run on a different computational supports, e.g. a Conventional PC, a RaspberryPi or Arduino.

Requirement analysis

We shall clarify the meaning of the names and of the verbs in the requirements:
  • button: a small device on a piece of electrical or electronic equipment which is pressed to operate it. It could be on every devices.
  • human user able: it must to be accesible to human's abilities
  • led: a light-emitting diode (a semiconductor diode which glows when a voltage is applied).
  • different computational supports: supports are not specify, it is important to underline that they are different so they need to comunicate each other.

    • Users Story

    When the user push the button present in a support, the led switch its state: if it is was on it will become off and the other way around.

    Case Study

    When the button is pressed we need an application that is listening mode. The messages from the button should be like an event for the possible application. Then the information that the button was pressed should be send to the led and it must shuld understand what it hve to do.     Possible Tecnologies

    The main problems are:
  • Comunication between the applications
    Distribuited systems need comunicate with each other, we should define how this could happened. A possible solution could be a RESTFUL Api like was implemented in this link RobotApi2021. Our situation is Human-to-Machine interaction using request-response pattern. Another possible solution could be use a support for WebSocket which interact between two application. But how it was specify in the case study button Listener perceives the message as an event, so the request response isn't able to implement that, they implement synchronous messages, they should be asynchronous It must be listeneing in a port whent it start comunication until the application is started. Event messages could be implemented through WebSockets support as they was implemented in ActorWithKotlinSupport.html or thourght using QActors. Anyway QActors support protocols like CoAP, as an example of a lightweight REST protocol suited for M2M interaction.
  • Tecnology indipendent
    The support devices aren't specify by the customer, our implementation shoud be focus to designe tecnologies indipendent software as most is possible. It introduce an requirements about the specialitation of the application and a possible Logical architeture where the tasks are appropriately assigned.
